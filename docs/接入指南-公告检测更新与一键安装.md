# 接入指南：公告、检测更新与一键安装

## 1. 整体流程

```text
管理台上传 APK
  → 后端验证并计算 SHA-256
  → 管理员发布版本
  → 客户端按 versionCode 检测更新
  → 用户点击“一键下载更新”
  → 客户端显示实时进度
  → 校验大小与 SHA-256
  → 通过 FileProvider 打开系统安装器
```

公告使用同一后端：管理员发布后，客户端“关于 → 公告”读取公开公告接口。

## 2. 后端部署接入

### 2.1 持久化目录

Docker Compose 已挂载：

```yaml
volumes:
  - classing-releases:/data/releases
```

原生部署应预先创建目录，并保证运行用户可写：

```bash
sudo install -d -o classing -g classing -m 0750 /var/lib/classing/releases
```

```dotenv
RELEASE_STORAGE_DIR=/var/lib/classing/releases
MAX_RELEASE_ARTIFACT_BYTES=262144000
```

### 2.2 Nginx

APK 上传通常超过默认限制，至少设置：

```nginx
client_max_body_size 260m;
proxy_read_timeout 300s;
proxy_send_timeout 300s;
```

修改后执行 `nginx -t` 并 reload。

### 2.3 数据迁移

服务启动时自动创建：

- `announcements`
- `app_releases`
- 对应公共查询索引

APK 本体不写数据库；数据库只保存文件名、大小、SHA-256 和发布元数据。

## 3. 管理员发布步骤

1. 登录 Classing Service Console。
2. 打开“公告与版本”。
3. 选择平台、渠道，填写 `versionName` 和递增的 `versionCode`。
4. 选择签名后的 APK。
5. 填写更新说明；需要立即开放时勾选“上传后立即发布”。
6. 上传完成后确认状态为 `PUBLISHED`。
7. 使用页面中的下载按钮抽查文件。

同平台、同渠道的 `versionCode` 不可重复。Android 更新包必须使用与已安装客户端相同的 applicationId 和签名证书，否则系统安装器会拒绝覆盖安装。

## 4. Android 客户端接入

### 4.1 公告

客户端调用：

```text
GET /api/v1/client/announcements?platform=ANDROID_MOBILE
```

按后端顺序展示标题与内容；网络失败时只显示本地化错误，不暴露后端内部响应。

### 4.2 版本检测

客户端携带 `BuildConfig.VERSION_CODE`：

```text
GET /api/v1/client/releases/latest?platform=ANDROID_MOBILE&channel=STABLE&versionCode=<current>
```

`updateAvailable = true` 时展示版本、文件大小、强制更新标记和 changelog。

### 4.3 下载与进度

客户端下载 `downloadUrl`，按响应流累计字节：

```text
progress = downloadedBytes / totalBytes
```

界面显示百分比、已下载大小和总大小。下载先写入 `cacheDir/updates/*.part`；网络中断时保留部分文件，下次使用 `Range: bytes=<已下载>-` 续传。完整大小或 SHA-256 校验失败时删除损坏的 `.part`，成功后再原子改名为 `.apk`。

### 4.4 完整性校验

下载完成必须满足：

- 实际字节数等于 `artifactSize`；
- 本地 SHA-256 等于响应中的 `sha256`。

任一校验失败都不得打开安装器。

### 4.5 安装器与权限

客户端 Manifest 包含：

```xml
<uses-permission android:name="android.permission.REQUEST_INSTALL_PACKAGES" />
```

APK 通过不可导出的 `FileProvider` 暴露给系统安装器。Android 8+ 未允许“安装未知应用”时，客户端先打开本应用的授权页；用户返回后再次点击“安装更新”即可继续。

## 5. 验证清单

- 公告创建后，公开接口只返回当前有效公告。
- 草稿版本不会被客户端检测到。
- 发布版本后，低 `versionCode` 客户端得到 `updateAvailable = true`。
- APK 支持完整下载和 Range 下载。
- 客户端下载进度从 0 到 100%。
- 篡改 APK 后 SHA-256 校验失败，安装器不会打开。
- 使用相同签名的 APK 可以覆盖安装。
- 删除版本后，数据库记录和 APK 文件同时消失。

## 6. 通道、限流与会话失效

- Mobile 关于页持久保存更新通道，默认 `STABLE`，用户可切换 `BETA`；每次 latest 查询都显式传递所选 `channel`，并展示响应中的实际通道。
- 公告、latest 和下载遇到 429 时解析 `Retry-After`，在冷却结束前禁止重复请求并显示剩余秒数。
- 账户和官方云请求统一解析响应 `code` 与 `Retry-After`。收到 `AUTH_SESSION_REVOKED`、`AUTH_REFRESH_REVOKED` 或旧会话导致的 401 时清理本地 token 并引导重新登录。
- 强制更新由 `forceUpdate`、`mandatory` 和 `minSupportedVersionCode` 联合决定；Stable/Beta 使用相同的容量检查、Range、长度、SHA-256 与安装流程。
