# 客户端账户与每日简报联调

本文记录 Android 客户端与当前后端实现之间的真实调用逻辑。正式基址固定为 `https://api-classing.underflo.ink`，请求和响应均使用 UTF-8 JSON。

## 1. 登录与会话

### 登录

`POST /api/v1/auth/login`

```json
{
  "identifier": "alice@example.com",
  "password": "UserPass123!"
}
```

`identifier` 支持邮箱或用户名。成功响应：

```json
{
  "session": {
    "accessToken": "...",
    "refreshToken": "...",
    "accessExpiresAt": 1780000000000,
    "refreshExpiresAt": 1782500000000
  }
}
```

- access token 默认有效期 15 分钟。
- refresh token 默认有效期 30 天，并在每次刷新时一次性轮换；同一 token、IP 与 User-Agent 在 5 秒内的并发请求会收到完全相同的 replacement session。
- 客户端应在 access token 剩余不足 60 秒时调用 `POST /api/v1/auth/refresh`。
- 客户端必须用 single-flight 串行化刷新；仅在服务端明确返回 `401 AUTH_REFRESH_REVOKED` 且本地仍保存本次尝试的旧 token 时清除会话，网络错误、超时和 5xx 不得清除新会话。
- Android 客户端的 HTTP 调用必须在后台线程执行，不得在 Compose 主线程直接调用同步网络 API。

### 当前账户与会员

登录后依次请求：

1. `GET /api/v1/account/me`
2. `GET /api/v1/membership/status`

两者均要求 `Authorization: Bearer <accessToken>`。客户端把返回结果合并为账户摘要；Wear 默认接收手机下发的只读摘要，自动状态不可用时也可经短时二维码获得独立加密会话，详见 `API-Wear二维码登录.md`。

## 2. 注册

`POST /api/v1/auth/register`

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "password": "UserPass123!"
}
```

- 用户名：3–40 位 ASCII 字母、数字、点、短横线或下划线。
- 密码：8–128 个字符。
- 用户名和邮箱全局唯一。
- 成功后直接返回与登录相同的 `session`。
- 客户端将注册放在账户页的二级页面；成功后返回账户主页。

## 3. 密码重置

### 申请

`POST /api/v1/auth/password/reset/request`

```json
{
  "email": "alice@example.com"
}
```

接口始终返回 `202` 和泛化文案，避免泄露邮箱是否存在。账户存在时，后端会：

1. 生成一次性 token，只在 `password_reset_tokens` 中保存哈希；
2. 创建 `PASSWORD_RESET` 邮件任务；
3. 由 SMTP 邮箱池异步投递 token 和到期时间；
4. 生产环境不在接口响应中返回 token。

开发或测试环境仅在 `EXPOSE_RESET_TOKEN=true` 时返回 `devResetToken`。

### 确认

`POST /api/v1/auth/password/reset/confirm`

```json
{
  "token": "email-one-time-token",
  "newPassword": "NewPass123!"
}
```

成功后 token 被标记为已使用、密码哈希更新、`auth_epoch` 提升，并撤销该用户全部 access/refresh 会话。客户端应清除本地 token，返回登录入口。

## 4. 会员兑换

`POST /api/v1/membership/redeem`

```json
{
  "code": "CLASSING-..."
}
```

兑换成功后客户端重新读取会员状态；官方云同步仅在已登录且会员有效时启用。

## 5. 每日简报

### 保存或更新

`PUT /api/v1/briefings/daily`

```json
{
  "enabled": true,
  "channel": "BOTH",
  "time": "20:00",
  "timezone": "Asia/Shanghai"
}
```

- `channel`：`APP_NOTIFICATION`、`EMAIL`、`BOTH`。
- `time` 必须是有效的 24 小时制 `HH:mm`，例如 `20:00`；`99:99` 会返回 `400 BRIEFING_INVALID`。
- `timezone` 必须是可加载的 IANA 时区；缺省为 `Asia/Shanghai`。
- `APP_NOTIFICATION` 由 Android 本机调度，可离线工作。
- `EMAIL` 和 `BOTH` 要求登录；后端仅为这两种通道生成邮件任务。

Android 12+ 若没有“精确闹钟”特殊权限，客户端会自动降级为非精确闹钟，不得因此崩溃。Android 13+ 的通知权限仍需由用户授权。

### 查询、取消与测试

- `GET /api/v1/briefings/daily`：查询当前订阅。
- `DELETE /api/v1/briefings/daily`：删除订阅，成功返回 `204`。
- `POST /api/v1/briefings/daily/test`：创建测试邮件任务，成功返回 `202`。

## 6. 错误结构与客户端展示

```json
{
  "code": "AUTH_INVALID_CREDENTIALS",
  "message": "identifier or password is incorrect",
  "requestId": "req_..."
}
```

客户端应使用 `code` 映射本地化文案，不直接把后端英文 `message` 作为主要界面提示。常见错误码：

- `AUTH_INVALID_CREDENTIALS`
- `AUTH_ACCOUNT_DISABLED`
- `AUTH_ACCOUNT_EXISTS`
- `AUTH_RATE_LIMITED`
- `AUTH_REFRESH_REVOKED`
- `AUTH_RESET_TOKEN_INVALID`
- `MEMBERSHIP_REDEEM_INVALID`
- `MEMBERSHIP_REDEEM_CONFLICT`
- `BRIEFING_INVALID`
- `BRIEFING_DISABLED`

## 7. 部署依赖

- 密码重置和邮件简报共用 `mailboxes` SMTP 邮箱池。
- 邮箱密码必须通过 `env:VARIABLE_NAME` 引用环境变量，不写入数据库。
- 必须启用 `SCHEDULER_ENABLED=true`，否则邮件任务不会投递。
- `CLASSING_SMTP_PASSWORD` 只是示例变量名，实际需与邮箱记录的 `passwordSecretRef` 一致。
